Помимо описания связей, в модели вынесены бизнес-методы, инкапсулирующие операции, неотделимые от состояния сущности. На листинге~\ref{lst:wallet_deposit} приведен метод пополнения кошелька абонента, обеспечивающий целостность изменения остатка и создания соответствующей транзакции в одном вызове.

\begin{lstlisting}[language=PHP,caption={Метод пополнения кошелька \texttt{Wallet::deposit}},label={lst:wallet_deposit}]
public function deposit($amount, $description = "Recarga de saldo")
{
    $this->balance += $amount;
    $this->save();

    $this->transactions()->create([
        'type'        => 'deposit',
        'amount'      => $amount,
        'description' => $description,
        'reference'   => 'DEP_' . uniqid(),
        'status'      => 'completed',
    ]);

    return $this;
}
\end{lstlisting}

Аналогичным образом метод \texttt{Wallet::makePayment} проверяет достаточность остатка перед списанием, исключая возможность отрицательного баланса. В модели \texttt{Invoice} вынесен статический метод \texttt{generateInvoiceNumber}, формирующий идентификатор счета вида \texttt{<<код заведения>>-<<код пункта эмиссии>>-<<сквозной номер>>} в соответствии с требованиями фискальной системы Эквадора (SRI). Тринадцать классов моделей используют трейт \texttt{Auditable}, что обеспечивает автоматическую запись изменений в таблицу \dbfield{audits} без необходимости явных вызовов в контроллерах. Дополнительно модель \texttt{Invoice} использует трейт \texttt{SoftDeletes}, сохраняющий удаленные счета в архивной выборке для целей аудита и фискальной отчетности.

Все контроллеры реализованы по единому шаблону, соответствующему стандартному набору действий ресурса (\texttt{index}, \texttt{show}, \texttt{store}, \texttt{update}, \texttt{destroy}). При необходимости стандартный набор расширяется специализированными действиями. В качестве примера на листинге~\ref{lst:invoice_controller} приведена сигнатура контроллера \texttt{Admin\textbackslash InvoiceController}, реализующего стандартный CRUD-набор и два дополнительных действия для выпуска счетов и фиксации оплаты.

\begin{lstlisting}[language=PHP,caption={Сокращенная сигнатура контроллера \texttt{Admin\textbackslash InvoiceController}},label={lst:invoice_controller}]
class InvoiceController extends Controller
{
    public function index(Request $request);
    public function show(Invoice $invoice);
    public function store(StoreInvoiceRequest $request, SettingService $settings);
    public function update(UpdateInvoiceRequest $request, Invoice $invoice);
    public function destroy(Invoice $invoice);

    public function configCheck(InvoiceConfigValidator $validator);
    public function generateAuto(AutoBillingService $billingService, InvoiceConfigValidator $validator);
    public function generateByContract(Request $request, AutoBillingService $billingService);
    public function charge(Request $request, Invoice $invoice, AutoBillingService $billingService);
}
\end{lstlisting}

Контроллер не содержит бизнес-логики напрямую: операции делегируются сервисам предметной области, внедряемым контейнером Laravel через типизацию параметров метода. Так, метод \texttt{generateAuto} перед запуском вызывает \texttt{InvoiceConfigValidator} для проверки полноты фискальных настроек и при успехе обращается к \texttt{AutoBillingService}. Валидация входных данных вынесена в классы запросов (\texttt{StoreInvoiceRequest}, \texttt{UpdateInvoiceRequest}), что упрощает повторное использование правил и автоматическое формирование сообщений об ошибках.

Расписание задач не закреплено жестко в исходном коде, а формируется динамически в файле \route{routes/console.php} на основе таблицы \dbfield{automation\_settings}. Сервис \texttt{AutomationSettingsService::applySchedule} читает действующие параметры (выражение cron, признак включения, тестовый режим) и регистрирует соответствующие записи в планировщике Laravel. Это позволяет администратору изменять частоту выполнения автоматизации непосредственно из панели управления без правки исходного кода. На листинге~\ref{lst:auto_billing_job} приведен фрагмент обработчика задачи \texttt{ProcessAutoBilling}, демонстрирующий принудительный сброс кэша настроек и проверку признака включения автоматизации при каждом запуске.

\begin{lstlisting}[language=PHP,caption={Фрагмент обработчика задачи \texttt{ProcessAutoBilling}},label={lst:auto_billing_job}]
public function handle(AutoBillingService $billing): void
{
    AutomationSetting::flushCache();
    $automation = AutomationSetting::getCached('auto_billing');

    $isEnabled = (bool) ($automation->enabled ?? false);
    if (!$isEnabled) {
        Log::info('ProcessAutoBilling skipped: disabled by automation_settings');
        return;
    }

    $report = $billing->processCompleteBilling();
    Log::info('ProcessAutoBilling finished', $report);
}
\end{lstlisting}

Покрытие тестами обеспечивается набором \route{tests/Feature/Jobs/}, который проверяет фактическую отработку каждой задачи в условиях временной базы данных SQLite в памяти.

Применяются как стандартные обработчики Laravel, так и разработанные в рамках настоящей работы. Перечень обработчиков и их роль приведены в таблице~\ref{tab:server_middleware}.

Обработчик \texttt{CheckPermission} реализует ядро ролевой модели разграничения доступа. Получив параметр <<идентификатор разрешения>> из объявления маршрута, он находит аутентифицированного сотрудника, при необходимости подгружает связанную роль вместе с набором разрешений и проверяет, входит ли требуемый идентификатор в этот набор. Сотрудники с ролью \texttt{super\_admin} пропускают любые проверки без обращения к базе данных, что снимает необходимость явного перечисления всех разрешений для этой роли. Объявление маршрута в \route{routes/api.php} имеет следующий вид:

\begin{lstlisting}[language=PHP,caption={Объявление маршрута с проверкой разрешения},label={lst:route_permission}]
Route::put('/clientes/{id}', [ClientController::class, 'update'])
    ->middleware('permission:clientes.editar');
Route::post('/roles/{id}/permissions', [RoleController::class, 'syncPermissions'])
    ->middleware('super_admin');
\end{lstlisting}

В системе принята политика умолчательного запрета: маршрут считается недоступным до тех пор, пока для него явно не задан соответствующий обработчик \texttt{permission} или \texttt{super\_admin}. Это исключает риск случайной открытой публикации новой конечной точки.

В дополнение к промежуточным обработчикам, валидация входных данных вынесена в самостоятельные классы запросов (\route{app/Http/Requests/}). Контроллер не принимает решений о пригодности данных~--~если правила нарушены, фреймворк автоматически формирует HTTP-ответ \texttt{422 Unprocessable Entity} со списком ошибок до фактического вызова метода контроллера. Подобное разделение существенно упрощает повторное использование правил валидации и автоматическое формирование документации программного интерфейса.

Размеры контейнера определяются динамически из родительского блока, благодаря чему график корректно адаптируется при изменении размеров окна.

Все модальные окна реализованы по единому шаблону: содержимое размещается в портале верхнего уровня документа, что исключает наложение проблем z-index, а взаимодействие с клавиатурой (закрытие по \texttt{Escape}, переход по \texttt{Tab}) обрабатывается централизованно.

\subsubsection{Слой обращения к программному интерфейсу}

Обращения к программному интерфейсу серверной части сосредоточены в каталоге \route{src/lib/api}. Модули каталога сгруппированы по доменам: \texttt{automations.ts}, \texttt{firewall.ts}, \texttt{mikrotik-routers.ts}, \texttt{notifications.ts}, \texttt{whitelist.ts}. Подобное разделение позволяет компонентам импортировать только необходимые функции и упрощает мокирование вызовов при модульном тестировании.

Базовый адрес сервера читается из переменной окружения \texttt{PUBLIC\_API\_BASE} (модуль \route{src/lib/config}). Каждый модуль реализует локальную функцию \texttt{authHeaders}, формирующую заголовки запроса с маркером доступа из локального хранилища браузера. На листинге~\ref{lst:auth_headers} приведен типичный фрагмент:

\begin{lstlisting}[language=Java,caption={Формирование заголовков с маркером Sanctum},label={lst:auth_headers}]
function authHeaders(): HeadersInit {
    const token = typeof window !== 'undefined'
        ? localStorage.getItem('employee_token')
        : null;
    return {
        Authorization: `Bearer ${token ?? ''}`,
        Accept: 'application/json',
        'Content-Type': 'application/json',
    };
}
\end{lstlisting}

Возвращаемые данные типизированы посредством общих интерфейсов TypeScript, описанных в \route{src/lib/types/}. Так, операции с правилами брандмауэра принимают и возвращают объекты типов \texttt{FirewallSnapshot} и \texttt{ApplyLog}, что исключает рассогласование форматов между клиентом и сервером.

Для устойчивости к временным сетевым сбоям применяется обертка с повторными попытками и экспоненциальной задержкой: запрос повторяется до трех раз с удвоением интервала при возникновении ошибок класса 5xx и сетевых таймаутов. Запросы, изменяющие состояние (методы \texttt{POST}, \texttt{PUT}, \texttt{PATCH}, \texttt{DELETE}), не повторяются автоматически~--~повтор инициируется только пользователем после явного подтверждения. Сводные показатели, обновляемые редко (структура каталога уведомлений, общесистемные настройки), кэшируются в \texttt{localStorage} с пометкой времени получения и принудительной инвалидацией через 24~часа.

\subsubsection{Уведомления пользователя}

Уведомления о результатах операций отображаются всплывающими карточками библиотеки \texttt{svelte-sonner}: успешное завершение длительных действий (применение правил брандмауэра, синхронизация очередей, выпуск партии счетов) подтверждается зеленой карточкой, ошибки и предупреждения~--~красной и желтой соответственно. Карточки автоматически скрываются по таймауту и поддерживают ручное закрытие. Подобный подход обеспечивает ненавязчивое информирование сотрудника без блокирования основного потока работы модальными окнами.

\subsubsection{Интеграционное тестирование RouterOS API}

Интеграция с маршрутизатором MikroTik проверяется на стенде, состоящем из одного устройства MikroTik (для разработки использован виртуальный экземпляр Cloud Hosted Router~7 и аппаратный маршрутизатор RB951Ui-2HnD с RouterOS~7.x) и серверной машины, выполняющей приложение. Доступ к интерфейсу API маршрутизатора осуществляется по TCP-портам 8728 (без шифрования) или 8729 (с шифрованием TLS), в зависимости от настроек устройства.

Базовая проверка подключения выполняется командой \route{php artisan mikrotik:test}, реализованной в классе \texttt{TestMikroTikConnection}. Команда выполняет следующую последовательность действий:
\begin{enumerate}
\item Чтение записи основного маршрутизатора из таблицы \dbfield{mikrotik\_routers}.
\item Установление подключения к API с использованием сохраненных параметров (хост, порт, имя пользователя, пароль).
\item Выполнение запроса \route{/system/resource/print} для подтверждения работоспособности соединения и получения сведений об устройстве (модель, версия RouterOS, время непрерывной работы).
\item Опциональная проверка прав на запись в подсистему очередей (флаг \texttt{-{}-write-test}): создание тестовой очереди в \route{/queue/simple} с уникальным префиксом и последующее ее удаление.
\item Обработка ошибок с детальной диагностикой: рекомендации по проверке доступности API, корректности учетных данных и сетевой связности.
\end{enumerate}

Для тестирования сценариев работы с правилами брандмауэра используется отдельная подсистема снимков: перед изменением фиксируется текущее состояние правил, после применения~--~записывается журнал применения в таблицу \dbfield{firewall\_apply\_logs}. При обнаружении ошибки или по явной команде сотрудника возможно восстановление предыдущего состояния путем обратного применения снимка. Это позволяет выполнять интеграционные тесты на действующем маршрутизаторе без риска необратимого нарушения его конфигурации.

Поскольку взаимодействие с реальным маршрутизатором требует наличия физического стенда, в основной автоматизированной выборке тестов клиентская библиотека \texttt{routeros-api-php} заменяется макетом Mockery. Тесты с реальным маршрутизатором запускаются отдельно по флагу окружения \texttt{MIKROTIK\_INTEGRATION\_TESTS=1} и не входят в обязательный набор для каждой сборки.
